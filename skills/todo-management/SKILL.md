# TODO Management

## Use When
- User says "todo", "task", "할 일", "투두", "이번주 할 일", "오늘 할 일"
- User wants to add, check, migrate, or review tasks
- User wants to add or manage subtasks ("add subtask", "서브태스크")
- User wants to add a comment or work log to a task ("코멘트", "comment", "이력")
- User wants to view task history or progress notes
- User asks about backlog or weekly planning
- User wants to view or manage tasks by topic/project
- Sunday weekly review or new week setup

## Do Not Use When
- Jira ticket management -- use jira-master instead
- GitHub issue tracking -- use git-issue-master instead
- Calendar/schedule management -- use gog-calendar instead

## First-Time Setup

On first invocation, check workspace config:

1. Read `~/.claude/CLAUDE.md` for `todo-workspace` entry
2. **If not found**: ask user for path (default: `~/todo-workspace/`), then `Edit` to append to `~/.claude/CLAUDE.md`:
   ```
   ## Todo Workspace
   todo-workspace: /absolute/path/to/todo
   ```
   Do NOT proceed until Edit confirms success.
3. **If found**: use the configured path

Initialize directory if it doesn't exist:
- Copy `<skill_base_dir>/templates/weekly.md` as the weekly template
- Copy `<skill_base_dir>/templates/backlog.md` as `backlog.md`
- Copy `<skill_base_dir>/templates/topics.md` as `topics.md`

## File Structure

| File | Purpose |
|---|---|
| `YYYY-WXX.md` | Weekly file (ISO week, Sunday start). Days Sun~Sat, each with `### P0`, `### P1`, `### P2` |
| `backlog.md` | Unscheduled/deferred tasks (P1, P2, Ideas) |
| `topics.md` | Topic/project registry (Active, Archived) |
| `templates/weekly.md` | Template for new week creation |

### Week Number Calculation

ISO week numbering with **Sunday as week start**: if today is Sunday, it belongs to the **next** ISO week (since ISO uses Monday start). Compute algorithmically. Example: Sunday 2026-02-22 = W09 (not W08).

## Task Syntax

Each task is a single line with optional metadata fields:

```
- [ ] Task description @assignee #tag %topic ~estimate due-date [REF-ID]
```

| Field | Format | Example | Notes |
|---|---|---|---|
| Assignee | `@name` | `@alice` | |
| Tag | `#tag` | `#bug`, `#feat`, `#docs` | Task type |
| Topic | `%topic` | `%air-backend` | Project/area. One per task, kebab-case |
| Estimate | `~Nh` or `~Nd` | `~2h`, `~1d` | |
| Due date | `YYYY-MM-DD` | `2026-03-05` | |
| External ref | `[ID]` | `[JIRA-101]` | |
| Link | `[text](url)` | `[slack](https://...)` | |
| Migration source | `<- Day` | `<- Mon` | Auto-generated by Migrate |
| Comment | `> MM-DD HH:mm: text` | `> 02-26 14:30: 확인 완료` | Auto-generated by Add Comment (see Comments) |

## Subtasks

2-space indent under parent. `(done/total)` is display-only. See Display Format for examples.

| Rule | Detail |
|---|---|
| Max depth | **2 levels only** (parent -> child). No nested subtasks |
| Metadata | All fields except `%topic` (inherited from parent) |
| Progress | Auto-display `(done/total)` counter next to parent |
| Completion | Parent `[x]` only when **all** children are `[x]` or `[-]` |
| Blocked | If any child is `[!]`, parent is considered blocked |
| Migration | Parent migrates with **all** children together as a unit |
| Estimates | Parent `~estimate` = total budget. Child estimates = breakdown |
| Adding | New subtask appended at end of children list |
| Dropping | Dropping parent (`[-]`) drops all children. Dropping a child does not affect parent |

---

## Comments (Work Log)

Timestamped progress notes attached to a task. `>` prefix with 2-space indent, placed **after subtasks**. See Display Format for examples.

| Rule | Detail |
|---|---|
| Format | `> MM-DD HH:mm: comment text` (2-space indent) |
| Timestamp | Auto-generated (KST). `HH:mm` optional for manual entries |
| Order | **Newest first** -- new comments prepended above older ones |
| Placement | After all subtasks (if any), before next task |
| Migration | Comments migrate **with parent task** as a unit |
| Completion | Preserved on `[x]` and `[-]` tasks (visible in weekly review) |
| Backlog | Comments follow the task when moved to/from backlog |
| Max length | One comment = one line. Split long notes into multiple comments |
| No nesting | Comments cannot have sub-comments |

## Status Symbols

| File syntax | Display icon | Meaning |
|---|---|---|
| `- [ ]` | ⬜ | Open -- not started |
| `- [x]` | ✅ | Done -- completed |
| `- [-]` | ⛔ | Dropped -- cancelled |
| `- [>]` | ➡️ | Migrated -- moved to next day |
| `- [!]` | 🟡 | Blocked -- waiting on something |

---

## Priority Levels

| Level | Meaning | Max Tasks |
|---|---|---|
| **P0** | Must do today | 1~3 |
| **P1** | Should do today (important but flexible) | no limit |
| **P2** | Nice to have (do if time allows) | no limit |

**Ideas** bucket (backlog only): no priority; promote to P1/P2 before scheduling.

## Operations

### Simple Operations

| Operation | Steps |
|---|---|
| **Show Today** | Find weekly file for current date -> read day section -> display by priority with status counts, `(done/total)` for subtasks |
| **Add Task** | Parse input -> target day (default: today), priority (default: P1) -> append to section. Subtasks: create parent + indent children 2 spaces |
| **Add Subtask** | Find parent (must not be a subtask itself) -> append indented 2 spaces at end of children |
| **Add Comment** | Find task -> generate `MM-DD HH:mm` (KST) -> insert after subtasks, before older comments (newest first) |
| **Show Log** | Find task -> display comments oldest->newest (reverse stored order). No comments: "No comments on this task." |

### Complete Task
1. Find the task by description or line number
2. Branch by task type:
   - **Parent with subtasks**: check all children are `[x]` or `[-]` before allowing completion
     - If incomplete children exist: warn user and list them
     - User can force-complete (marks all open children `[-]`) or complete children first
   - **Subtask**: change `[ ]` to `[x]`, then check if all siblings are done to prompt parent completion
   - **Normal task** (no subtasks, not a subtask): change `[ ]` to `[x]`
3. Confirm completion

### Migrate Tasks (End of Day)
1. Find all `[ ]` (open) and `[!]` (blocked) parent-level tasks in today's section
2. For each task:
   - `[ ]` tasks: change to `[>]`, copy to next day as `[ ]` with `<- Day` marker
   - `[!]` tasks: migrate keeping `[!]` status (do not change to `[>]`), add `<- Day` marker
   - Subtasks: migrate with parent, keep their current status
   - Comments: copy as-is with parent
3. **3+ day migration rule**: prompt user -- move to backlog, drop `[-]`, or keep (must justify)
4. **Cross-week** (past Saturday): default to next week's Sunday. If 3-day rule triggers, offer backlog. Create next week file from template if needed.

### New Week Setup (Sunday)
1. Create new weekly file from template: `YYYY-WXX.md`
2. Calculate date range (Sun ~ Sat) and fill in headings
3. Pull tasks from backlog that have due dates in this week
4. Show pulled tasks for review

### Weekly Review (Saturday)
1. Read all `[>]` tasks from the week
2. For each, ask user to decide:
   - Move to next week's Sunday (with subtasks and comments)
   - Move to backlog (with subtasks and comments)
   - Drop with `[-]` (drops parent and all children; comments preserved for review)
3. Show weekly summary:
   - Total tasks: created, completed, dropped, migrated (parent-level only)
   - Completion rate
   - Per-topic breakdown (see Topic Operations -> Topic Summary)

## Backlog Operations

| Operation | Steps |
|---|---|
| **Show Backlog** | Read `backlog.md` -> display by P1/P2/Ideas with counts. Empty: "Backlog is empty." |
| **Add to Backlog** | Parse input -> priority (default: P2, Ideas for vague items) -> append with subtasks/comments |
| **Promote** | Find task in backlog -> move to weekly file day (default: today) / priority (default: P1) with subtasks/comments -> remove from backlog |
| **Review** (Sunday) | For each task: keep / promote / drop -> show summary |

## Topic Operations

Topics are cross-cutting filters, not containers. Tasks live in weekly/backlog files; `%topic` is a filter.

### List Topics
1. Read `topics.md`
2. Show active topics with task counts (scan current week + backlog for `%topic` matches)
3. Display format:
```
## Active Topics
- air-backend (3 open | 5 done)
- search-engine (1 open | 2 done)

## Archived
- auth-migration (completed 2026-02)
```

### Show Topic
1. Accept topic name (e.g., "show topic air-backend")
2. Scan current week file + backlog for tasks with matching `%topic`
3. Group by source: "This Week" (with day/priority info) and "Backlog"
4. Show subtasks, comments, estimate rollup, and blocked count
5. Display format:
```
## Topic: air-backend (3 open | 2 done)

### This Week (W09)

⬜ 1. Implement auth API (1/3)                              (Thu P0)
   └ @alice | #feat | ~4h | 2026-02-28 | [JIRA-101]
   └ [x] Define API spec ~1h
   └ [ ] Implement endpoints ~2h
   └ [ ] Write tests ~1h
   💬 02-26 14:30: API 스펙 리뷰 완료, v2 반영

✅ 2. Fix DB connection pool                                (Wed P1)
   └ #bug | ~1h
   💬 02-25 11:00: HikariCP max-pool 30으로 조정

### Backlog

⬜ 3. Add rate limiting
   └ #feat | ~3h

---
Estimate remaining: ~9h | Blocked: 0
```

### Topic Log
1. Accept topic name (e.g., "topic log 항공-내재화" or "%항공-내재화 이력")
2. Scan current week + backlog for tasks with matching `%topic` that have comments
3. Collect all comments, group by task, sort by timestamp (newest first)
4. Display as unified timeline:
```
## %항공-내재화 Timeline

02-26 14:30 | git repo 세팅
  아키텍처 리뷰 완료, Spring Boot 3.2 기반으로 결정

02-25 10:00 | git repo 세팅
  팀 미팅에서 초기 방향 논의함

02-24 16:00 | API 스펙 정의
  v1 스펙 초안 작성 완료, 리뷰 요청함

---
3 comments across 2 tasks
```

### Add Topic
1. Accept topic name and optional description
2. Validate kebab-case naming
3. Append to `## Active` section in `topics.md`
4. Confirm addition

### Archive Topic
1. Accept topic name
2. Scan current week + backlog for open tasks with `%topic`
3. If open tasks exist: warn user and list them, ask to confirm
4. Move topic from `## Active` to `## Archived` with completion date
5. Confirm archival

### Topic Summary (part of Weekly Review)
During Saturday weekly review, show per-topic breakdown:
```
### Topic Summary
| Topic | Created | Done | Dropped | Migrated | Rate |
|---|---|---|---|---|---|
| air-backend | 5 | 3 | 0 | 2 | 60% |
| search-engine | 2 | 2 | 0 | 0 | 100% |
```

## Display Format

When showing tasks, use this **clean list** format. Status icons from the Status Symbols table.

```
## Today: Wed 02-26 (W09)

### P0 (1/1)

✅ 1. Deploy hotfix
   └ #ops | ~1h | [JIRA-301]

### P1 (0/3)

⬜ 2. Review PR for auth module
   └ @alice | #review | ~2h

🟡 3. Update API docs
   └ #docs
   └ blocked: waiting on spec
   💬 02-26 09:00: 스펙 담당자에게 확인 요청함

⬜ 4. git repo 세팅 (1/4)
   └ @bob | %항공-내재화 | #feat | ~2d
   └ [x] 기본 아키텍처 설계 ~2h
   └ [ ] 패키지 구조 세팅 ~2h
   └ [ ] 라이브러리 선정 및 추가 ~1h
   └ [ ] 배포 스크립트 세팅 ~3h
   💬 02-26 14:30: 아키텍처 리뷰 완료, Spring Boot 3.2 결정
   💬 02-25 10:00: 팀 미팅에서 초기 방향 논의

### P2 (0/1)

⬜ 5. Clean up test fixtures
   └ #refactor | ~1h

---
1/5 done | 1 blocked | 0 migrated
```

**Rules:**
- Numbers are sequential across all priorities
- Footer counts are parent tasks only; `(done/total)` inline counter is subtask progress and does not affect footer

## Error Handling

| Scenario | Action |
|---|---|
| Weekly file not found | Offer to create from template |
| Workspace not configured | Run first-time setup |
| No tasks for today | Show "No tasks for today. Add one?" |
| Backlog empty | "Backlog is empty." |
| Topic not in registry | Offer to add it to `topics.md` |
| No tasks for topic | "No tasks found for %topic-name." |
| Topics file not found | Create from template |
| Complete parent with open subtasks | Warn and list incomplete children |
| Add subtask to a subtask | Reject: "Max 2 levels. Subtasks cannot have children." |
| Subtask without parent | Treat as regular task (remove indent) |
| Add comment to non-existent task | "Task not found. Check task description or use 'show today' to see tasks." |
| Show log with no comments | "No comments on this task." |
| Topic log with no comments | "No comments found for %topic-name." |
| 3-day migration: invalid response | Re-prompt with the three options (backlog, drop, keep) |
| Archive topic with open tasks confirmed | Move topic to Archived; open tasks keep their `%topic` tag as-is |
| Blocked task at end-of-day | Migrate keeping `[!]` status (see Migrate Tasks) |
